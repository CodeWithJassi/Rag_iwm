     Context

     Three complementary fixes to address retrieval failures (Q9, Q10, Q20):

     1. Structure-aware splitting: The current chunkers are length-first (split
     at ~2100 chars, use
     structure only as a hint). Bold **Label:** lead-ins in earnings-call
     sections вҖ” which are the
     document's own topic boundaries вҖ” are ignored. A new "structure" strategy
     makes each labeled
     block its own chunk.
     2. Contextual chunk prefix: Chunks are embedded as bare text. Two chunks
     that both mention
     "decline of ~6%" from different sections embed identically. Prepending
     company + section label
     at embed time makes the embedding carry disambiguating context.
     3. Prompt disambiguation: When multiple chunks with similar figures are
     retrieved, the LLM
     isn't told to distinguish them by section/period.

     These three changes reinforce each other: 1 creates correctly-bounded
     chunks, 2 makes them
     embed distinctly, and 3 tells the LLM to use the distinctions.

     Matches pymupdf4llm's markdown output for lines like **Future segment

     Algorithm:
     1. Walk lines. When _BOLD_LABEL matches вҶ’ flush current block, start new
     one with

     unlabeled

     3. Floor pass: merge any block whose len(content) < floor into the PREVIOUS
     block.

     4. Ceiling pass: split any block whose len(content) > ceiling using
     _split_page_char(), preserving the section label on each sub-chunk.

     Dispatcher change in retrieval_chunks(): add "structure" to the strategy
     if/elif chain, calling _split_page_structure(text, STRUCTURE_FLOOR_CHARS,

     After chunks = retrieval_chunks(pages) and before embedding, assemble a
     prefix per chunk:

     def _prefix(chunk, company):
         parts = [company]
         if chunk.get("section"):
             parts.append(chunk["section"])
         return " вҖ” ".join(parts) + ": "

     prefixed = [_prefix(c, company) + c["content"] for c in chunks]
     vectors = embed_documents(prefixed)

     The DB still stores c["content"] (no prefix in chunks.content вҖ” invariant
     #5). The
     prefix only exists in the embedding input and is never persisted.

     The _prefix() helper is a module-level function so it's easy to enrich
     later (add broker

  what it did find.

  Then retrieval.py line 343 checks a.startswith("INSUFFICIENT") and discards
  everything. Useful information is thrown away.

  Root cause: Two things colliding:

  1. The SUB_ANSWER_PROMPT says "If the passages do not contain the answer, 
  reply exactly: INSUFFICIENT" вҖ” but doesn't define "the answer." When the
  question has two facets and only one is found, the model doesn't know whether
  to answer partially or abstain. It tries to do both: starts with INSUFFICIENT,
  then gives what it found.
  2. The startswith("INSUFFICIENT") check in retrieval.py is too blunt. It
  treats any response that begins with the word as a total failure, even when
  the response goes on to provide extractable facts.

  Solution: Two changes needed:

  - Prompt: Clarify that partial answers are acceptable вҖ” if some facets are
  found and others aren't, answer what you can and note the gap. Don't say
  INSUFFICIENT unless truly nothing relevant was found.
  - Code: Change the check from startswith("INSUFFICIENT") to exact-match or a
  tighter guard (e.g. strip() == "INSUFFICIENT"), so a response that contains
  INSUFFICIENT as a preamble but then gives real content isn't discarded.

  ---
  Problem 2: GLM works in deep mode but not normal mode; Gemma fails in both

  The question asks for two numbers (domestic decline % + export decline %). In
  normal mode, this is one sub-question asking for both. That's high cognitive
  load вҖ” the model has to scan chunks, find two distinct numbers, and report
  both. Weaker models (Gemma, Llama) get overwhelmed and abstain.

  In deep mode, the decompose step splits it into two simple sub-questions:
  "domestic decline?" and "export decline?". Each is a single-number lookup.
  Even though the dedup guard may merge near-duplicates back, when decomposition
  succeeds at producing two genuinely scoped sub-questions, each one is easier.
  GLM вҖ” being the strongest of the three models вҖ” can handle the simpler
  sub-question in deep mode but fails on the compound question in normal mode.

  Gemma fails in both because it's not strong enough to extract the number even
  from a simpler sub-question вҖ” it sees the chunk about off-highway but can't
  pull the ~6% out of the dense financial text.

  Root cause: A compound question ("both domestically and in exports") asked as
  a single sub-question is too hard for most local models. Decomposition into
  separate sub-questions makes each one tractable for stronger models, but
  weaker models still struggle with number extraction from dense prose.

  Solution:

  - The dedup guard currently merges near-duplicate sub-questions back into the
  standalone question, which defeats decomposition for exactly this pattern. The
  dedup should be smarter: instead of merging to the standalone, keep the split

  words are meaningful dimension words.
  - Or, go the other direction: accept that compound questions are hard and let
  the SYNTHESIZE_PROMPT handle partial results gracefully (the fix we already

в—Ҹ All clean. Here's the full list of what was slowing things down and what
  changed:

  ---
  Four bottlenecks fixed

  1. Sequential sub-question loops вҶ’ Parallel (retrieval.py)

  Was: Retrieval and sub-answer generation ran one-at-a-time. With 2
  sub-questions in deep mode, that's retrieve вҶ’ wait вҶ’ retrieve вҶ’ wait вҶ’ answer 
  вҶ’ wait вҶ’ answer вҶ’ wait. Each wait is an LLM call (~30-50s).

  Now: All retrievals run in parallel via ThreadPoolExecutor, then all
  sub-answers run in parallel. With 2 sub-questions, wall-clock time is roughly
  halved вҖ” both retrievals complete in ~one retrieval's time, both sub-answers
  in ~one generation's time.

  Thread safety: Added _trace_lock for the trace list, _gate_lock for the
  gated-out list. Python dict key assignment is atomic under the GIL for
  non-conflicting keys, so retrieved[sq] and sub_answers[sq] are safe without
  locks.

  2. LLM_CYCLES: 2 вҶ’ 1 (config.py)

  Was: Every LLM call retried the entire provider chain a second time. With only
  local Ollama as provider, this meant every failed call got one pointless
  retry of the same endpoint.

  Now: One pass through the provider chain. The _try_ollama function already has
  its own retry for the specific "model loading" case, so cycle-level retry was
  redundant for the single-provider default.

  3. Embedding timeout: 300s вҶ’ 30-120s (embeddings.py)

  Was: Every embedding HTTP request had a 5-minute timeout. If the cluster was
  unreachable, the system hung for 5 minutes before falling back to local.

  Now: Three tiers:
  - Single query embedding (embed_query): 30s timeout вҖ” a single text embedding
  should never take longer
  - Batch ingestion (many chunks): 120s timeout вҖ” GPU batches need more time
  - Default (_embed_raw): 60s

  4. LLM timeout: 300s вҶ’ 180s (llm.py)

  Was: Ollama HTTP calls had 5-minute timeout. A stuck generation would block
  for 5 minutes.

  Now: 180s (3 minutes) вҖ” still safe for the slowest local model, but won't hang
  indefinitely.

  ---
  Round 2: Four more optimizations

  1. Batch query embeddings (retrieval.py)

  Was: In retrieve_for(), each rephrasing variant called embed_query()
  separately вҖ” 3 separate HTTP round-trips to Ollama in deep mode.

  Now: embed_queries_batch() sends all variants in a single /api/embed call.
  Ollama processes them as one batch on the GPU. 3 HTTP calls вҶ’ 1. Saves ~1-2s
  per sub-question.

  vector_search() gained an optional vec= parameter вҖ” when provided, the
  embedding call is skipped entirely.

  2. Reranker preload at startup (rerank.py + main.py)

  Was: The bge-reranker-v2-m3 model loaded lazily on the first query. Every
  restart of the server, the first user query paid a ~5s cold-start penalty.

  Now: rerank.preload() is called during FastAPI startup. The model loads before
  any query arrives. First query is as fast as the second.

  3. Tighter max_tokens (retrieval.py)
