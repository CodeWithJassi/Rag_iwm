  Plan: Table & Image Extraction for Ingestion Pipeline

  What already works in our favor

  The schema and pipeline already carry chunk_type вҖ” it's just hardcoded to

  ---
  Phase 1 вҖ” Table Extraction

  The problem

  Broker reports put critical data in tables: quarterly P&L, valuation
  multiples, peer comparisons, segment revenue splits. When tables are embedded
  in prose chunks via pymupdf4llm's markdown, the embedding model sees a wall of
  pipe characters вҖ” it can't "read" a table the way an analyst scans rows and
  columns. Dedicated chunk_type: "table" chunks with clean markdown tables let
  the retriever surface exact figures.

  Step 1.1: extract.py вҖ” extract tables from PDF

  Two sources, combined with deduplication:

  Source A вҖ” pdfplumber .extract_tables() (already a dependency):
  - Call page.extract_tables() on every page
  - Each table is a list[list[str | None]] вҖ” serialize to GitHub-flavored
  markdown (| col | col |)
  - Tag with page_no

  Source B вҖ” parse markdown tables from pymupdf4llm output:
  - pymupdf4llm already renders many tables as markdown in the page text
  - Scan each page for markdown table blocks (lines starting/ending with |)
  - Extract them as separate table entries

  Deduplication: A pdfplumber table and a pymupdf4llm markdown table on the same
  page with overlapping content should merge. Simple heuristic: if two tables
  on the same page share >50% of their numeric cells, keep only the cleaner one
  (prefer pdfplumber вҖ” it preserves cell alignment).

  Fallback: If neither source produces tables, that's fine вҖ” many pages have
  none.

  New function: extract_tables(path) -> list[dict] where each dict is
  {"page_no": int, "content": str (markdown table), "source": "pdfplumber" | 
  "pymupdf4llm"}.

  Step 1.2: chunking.py вҖ” convert tables to chunks

  table_chunks(tables: list[dict]) -> list[dict]:
  - Large markdown tables (over RETRIEVAL_CHUNK_CHARS) get split into sub-tables
  (keep the header row, split body rows)
  - Very small tables (<50 chars, like a 2Г—2) get dropped вҖ” they're probably
  page-number artifacts
  - Each chunk gets chunk_type: "table", page_no, section (from nearest heading
  on that page)
  - A prefix labels the chunk for embedding: "[TABLE] Page {n}" вҖ” helps the
  embedding model understand this is structured data, not prose

  Risk: duplication between text chunks and table chunks

  When pymupdf4llm renders a table as markdown, the same data appears in both:
  - A text chunk containing the paragraph around the table (with the markdown
  table inline)
  - A dedicated table chunk

  This isn't harmful вҖ” RRF fusion treats them as independent chunks, and having
  the same fact retrievable through both a prose query ("what was Q3 revenue?")
  and a structured query ("show me the quarterly P&L") is a feature. The
  embedding model reads both forms differently.

  If duplication becomes noisy in results, we can add a post-hoc dedup step:
  strip markdown tables from text chunks before embedding. Deferred until it
  proves to be a real problem.

  ---
  Other Ideas for Richer Chunking & Ingestion

  Here are additional ideas beyond tables. Each is scoped so you can pick and
  choose.

  A. Chunk importance scoring (low effort, high impact)

  Have the LLM rate each chunk 1вҖ“5 during ingestion. The score goes into
  metadata.importance. During retrieval, importance acts as a tiebreaker: when
  two chunks have similar cosine scores, the more important one ranks higher.

  Broker reports have natural importance gradients вҖ” the cover page (target
  price, rating) matters more than the disclaimer page. A static importance
  score lets the retriever exploit this without any runtime LLM cost.

  Cost: One extra field in the enrichment prompt (already making an LLM call if
  enrichment is on). Zero runtime overhead.

  B. Section breadcrumbs (trivial, no LLM needed)

  Currently section is just the nearest heading. Store the full heading path:
  {"section_path": ["Financial Performance", "Quarterly Trends", "Revenue"]}

  This is pure string processing вҖ” pymupdf4llm's markdown already has the
  heading hierarchy. The path helps the LLM understand where a chunk sits in the
  report's argument structure, and could be rendered in the UI as a breadcrumb
  trail in the source drawer.

  C. Numeric fact index (medium effort)

  For every chunk (not just tables), extract metric: value: unit triples:
  ["Revenue: 450: INR crore", "EBITDA margin: 26.7: %", "PAT growth: 12.0: %
  YoY"]

  Store in metadata.facts. This enables:
  - Fact-aware retrieval: a query for "EBITDA above 100 crore" could filter
  chunks that don't contain EBITDA at all
  - Fact highlighting in UI: the source drawer could bold the extracted numbers
  - Future: a structured query mode ("show me all chunks with revenue growth >
  10%")

  Cost: One extra field in the enrichment prompt. Validation needed to catch LLM
  hallucinations (did the LLM invent a number not in the text?).

  D. Chunk provenance tracking (small)

  Store which extractor produced the chunk in metadata:
  {"extractor": "pymupdf4llm", "chunking_strategy": "structure", "page_no": 5}

  When debugging retrieval quality issues, you can trace a bad answer back to
  exactly how its source chunk was created. Useful for A/B testing chunking
  strategies.

  E. Hypothetical answer embedding (HyDE for chunks)

  Instead of generating questions, generate a hypothetical dense paragraph that
  summarizes what the chunk is about, written in the style of an analyst's
  answer. This is the inverse of question generation вҖ” it makes the chunk look
  like the output rather than the input.

  A query embedding and a hypothetical-answer embedding are more directly
  comparable than a query embedding and a raw-chunk embedding.

  Cost: One more LLM call per chunk. Probably too expensive for every chunk, but
  worth it for high-value sections (executive summary, investment thesis,
  valuation).

  F. Table-aware summarization (deferred)

  Currently summarize_report() gets full_text(pages) вҖ” tables are embedded in
  the prose but not explicitly labeled. Adding a [TABLE] marker in the text
  before summarization would tell the summarization LLM "this is structured
  data, read it carefully." Small prompt engineering change, potentially big
  improvement in summary quality for numbers-heavy reports.
